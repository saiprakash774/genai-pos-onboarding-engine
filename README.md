# Master Architecture & Implementation Blueprint: GenAI Menu Onboarding Engine

This document details the transition of the GenAI Infor POS Menu Onboarding Engine from initial concept ("Casual Vibe Coding") to a production-ready, enterprise-grade system ("Disciplined Agentic Engineering"). The system is designed to ingest chaotic vendor spreadsheets, parse them using AI-driven logic, and output strictly validated JSON schemas for enterprise POS backends.

---

## I. Multi-Agent System Architecture

The core of the system relies on three specialized agents operating from a shared cross-tool foundation defined in the `agents.md` file.

| Agent Role | Primary Function | Technical Execution |
| --- | --- | --- |
| **Orchestrator Agent** | Pipeline management and user interaction. | Manages the front-end UI and coordinates tasks between specialized agents. |
| **Parsing Agent** | Data wrangling and contextual mapping. | Executes Python/Pandas scripts to unmerge cells, forward-fill missing parent data, and map modifiers. |
| **Validation Agent** | Enterprise gatekeeping and payload generation. | Verifies the output against Infor POS schema rules and standardizes the final JSON payload. |

---

## II. Phased Implementation Strategy

To build a reliable execution harness, the project follows a strict five-phase development lifecycle.

### Phase 1: Project Scaffolding

Establish the immutable directory structure to keep data, specifications, and procedural memory isolated.

* Create `/specs` for behavioral requirements.
* Create `/skills` for procedural agent memory.
* Create `/data/raw` for isolated data staging.
* Initialize the global `agents.md` configuration file.

### Phase 2: Refine Specifications into Gherkin/BDD Format

To enable Spec-Driven Development, all high-level business rules must be translated into behavioral specifications.

* Create `.feature` files (e.g., `specs/menu_mapping.feature`).
* Define exactly how the Parsing Agent should handle anomalies using the Given/When/Then format.
* Require that missing parent data traversing sizes (Tall, Grande, Venti) explicitly inherits the "Base_Drink" value via forward-fill logic.

### Phase 3: Build the Execution Harness & Detail Agent Skills

An agent's reliability is entirely dependent on its execution harness and memory management.

* **Configure a Sandbox (gVisor):** The Parsing Agent must execute Python/Pandas code in an isolated, ephemeral environment to prevent security risks or unintended side effects on the host system.
* **Implement Progressive Disclosure:** Flesh out the `excel_ingestion_skill.md` file using progressive disclosure. Start with a strict, minimal metadata footprint (approx. 50 tokens) describing exactly when to use the skill.
* **Prevent Context Rot:** Ensure complex spreadsheet manipulation instructions are only loaded into the agent's memory when a task matches that specific capability.

### Phase 4: Implement Interoperability via MCP

Enterprise integrations must be standardized, avoiding fragile, bespoke wrappers or hard-coded API scripts.

* Setup a Model Context Protocol (MCP) server for the Validation Agent.
* Utilize the MCP as a standardized, secure transport layer to fetch schema requirements and push JSON payloads to the Infor ION API.

### Phase 5: Establish Trajectory Evaluation

Correct final outputs can hide dangerously flawed reasoning. The path the AI takes to reach a conclusion must be fully auditable.

* Setup OpenTelemetry to trace the "Vibe Trajectory" of the agents.
* Verify the logical reasoning steps (e.g., ensuring the Mapping Agent linked "Cold Foam" to "Cold Coffee" strictly because of the `Applies_To` logic in the specification, rather than a lucky hallucination).

---

## III. Strict Coding Expectations

To maintain Disciplined Agentic Engineering, all generated code and system interactions must adhere to the following principles:

* **Code is Disposable:** Python scripts generated by the Parsing Agent are temporary tools. If a script fails the Gherkin BDD tests in Phase 2, the agent must discard the code and rewrite it. The specification is permanent; the code is not.
* **No Direct File Manipulation:** Agents must never overwrite raw data located in the `/data/raw` directory. All outputs must be distinctly routed to a `/data/output` directory.
* **Telemetry over Output:** System success is measured by the clarity of the OpenTelemetry trace, not just the compilation of a valid JSON file. Unexplained successful outputs will be treated as system failures.
* **Strict Sandbox Enforcement:** No data analysis or script execution may occur outside of the designated secure execution environment.
